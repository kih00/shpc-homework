1. 코드 설명

  이 프로젝트는 LFM2Model이라는 현대적인 하이브리드 트랜스포머 모델의 CPU 레퍼런스 구현체입니다.

   * 핵심: 파일 확장자는 .cu이고 Makefile에 nvcc 컴파일 설정이 있어 CUDA 프로젝트처럼 보이지만, 실제로는 CUDA 코드가 전혀 없으며 OpenMP를
     이용해 CPU 병렬 처리만 수행합니다.
   * 아키텍처: Attention (GQA), RMSNorm, RoPE, Sparse Mixture-of-Experts (MoE), Mamba 스타일의 1D Convolution 등 최신 기술들이 결합된
     복잡한 모델입니다.
   * 목적: GPU 가속 없이, 모델의 정확성과 로직을 검증하기 위한 순수 C++ 기반의 기준 구현으로 보입니다.

  2. 테스트 방법

  tests 디렉터리 안에 각 주요 구성 요소를 독립적으로 검증하기 위한 테스트 코드가 존재합니다.

   * 구조: tests/attn, tests/conv, tests/moe 와 같이 각 기능별로 하위 디렉터리가 나뉘어 있습니다.
   * 실행: 각 하위 디렉터리 (tests/attn 등)에 있는 Makefile을 사용하여 테스트를 빌드하고 run.sh 스크립트로 실행할 수 있습니다.
   * 목적: 이 테스트들은 현재의 CPU 구현이 올바르게 동작하는지 확인하는 것을 목표로 하며, 아마도 이곳에 실제 CUDA 커널의 레퍼런스 코드가
     있을 수 있습니다.

  3. 성능 측정 방법

  전체 모델의 성능은 루트 디렉터리의 Makefile과 run.sh를 통해 측정할 수 있습니다.

   1. 빌드: 프로젝트 루트 디렉터리에서 make 명령어를 실행하여 전체 모델을 빌드합니다.
   2. 실행: run.sh 스크립트를 실행합니다.
   3. 측정: src/main.cpp 파일의 로직에 따라, model.forward() 함수를 여러 번 실행하고 전체 소요 시간을 측정하여 초당 처리 샘플 수
      (throughput)를 계산하는 방식으로 성능이 측정됩니다.

  4. 성능 개선을 위한 핵심 영역

  현재 코드는 CPU 기반이므로, GPU를 활용하도록 변경하는 것이 성능 개선의 핵심입니다. 다음 두 파일을 집중적으로 수정해야 합니다.

   1. `src/layer.cu` (가장 중요)
       * 문제점: 이 파일은 행렬 곱셈(matmul), 정규화(rms_norm) 등 딥러닝의 핵심 연산을 담고 있습니다. 특히 matmul 함수가 매우 비효율적인
         3중 반복문으로 구현되어 있어 가장 큰 성능 병목 지점입니다.
       * 개선 방안: tensor_ops 네임스페이스의 함수들을 모두 실제 CUDA 커널로 재작성해야 합니다. matmul의 경우, cuBLAS 라이브러리를
         사용하는 것이 표준적인 고성능 구현 방법입니다.

   2. `src/model.cu`
       * 문제점: Attention, MLP, SparseMoeBlock 등 모델의 레이어들이 C++로 구현되어 있습니다. layer.cu의 연산들을 CUDA로 바꿔도, 이 파일의
         로직이 CPU에서 실행되면 CPU-GPU 간의 불필요한 데이터 전송이 발생하여 성능이 저하됩니다.
       * 개선 방안: forward 함수들을 포함한 각 레이어의 로직을 CUDA 커널을 호출하도록 변경하여, 전체 모델 추론 과정이 GPU 내에서 끝나도록
         만들어야 합니다.

  요약하자면, 이 프로젝트는 "CUDA로 포팅될 것을 상정하고 C++로 작성된 딥러닝 모델의 뼈대" 입니다. 성능을 개선하려면 src/layer.cu의 기본
  연산들을 CUDA로 구현하고, src/model.cu의 모델 로직을 GPU에서 동작하도록 수정하는 작업이 필요합니다.
